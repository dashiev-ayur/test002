# Что сделано: фаза 4

Документ фиксирует HTTP-слой MVP после фазы **4** (Backend API). Источник контрактов — [ТЗ.md](./ТЗ.md), порядок работ — [План реализации MVP.md](./План%20реализации%20MVP.md).

---

## Реализовано

### Три маршрута (как в ТЗ)

- `POST /shops/{shopId}/telegram/connect`
- `POST /shops/{shopId}/orders`
- `GET /shops/{shopId}/telegram/status`

### DTO и Validator

- **`TelegramConnectRequest`**, **`CreateShopOrderRequest`** — валидация тел запросов.
- После разбора JSON выполняется **`trim`** полей.
- Ошибки валидации → **422** с телом `{"errors": {...}}`.
- Некорректный JSON или пустое тело → **422** с `{"error": "..."}`.

### Исключения и HTTP-коды

- **`ShopNotFoundException`** → **404**.
- **`InvalidTelegramIntegrationDataException`** → **422** (защита от пустых полей после trim; смысл пересекается с валидатором, но сохраняется контракт прикладного сервиса).

### JSON-ответы

- **connect:** `id`, `enabled`, маскированные `botToken` / `chatId`, `createdAt`, `updatedAt` (ISO 8601).
- **заказ:** объект `order` + `notificationStatus` (`sent` / `failed` / `skipped`), код **201**.
- **статус:** `enabled`, маскированный `chatId` (или `null`, если интеграции нет), `lastSentAt` (последняя **успешная** отправка `SENT`), `sentCount` / `failedCount` за 7 суток.

### Сервисы и утилиты

- **`TelegramIntegrationStatusService`** — сбор данных для статуса и маскирование `chatId` для API.
- **`SensitiveDataMasker`** — единые правила маскирования для ответов.

### CORS

- Подписчик **`CorsSubscriber`** для путей `/shops/…`.
- Переменная **`CORS_ALLOW_ORIGIN`** в `.env` (по умолчанию `*`).
- В `config/services.yaml` — запасной дефолт через параметр **`default_cors_allow_origin`**, если переменная не задана.
- При значении `*` и наличии заголовка `Origin` в ответ подставляется этот origin (удобно для локального фронта на другом порту).

### Тесты

- **`tests/Integration/Phase4ShopApiSerializationTest.php`** — проверка десериализации тел без обращения к БД.
- Полный прогон интеграционных тестов с Doctrine по-прежнему требует созданной БД **`postgre_test`** (и миграций), как для фазы 3.
