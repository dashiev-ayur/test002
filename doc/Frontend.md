# План создания фронтенда

Документ фиксирует стек, архитектуру и порядок работ по экрану настроек Telegram (контракты с backend — в [ТЗ.md](./ТЗ.md)).

---

## 1. Технологический стек

- **Vite**
- **TypeScript**
- **React** — UI.
- **React Router** — маршрутизация; корневой маршрут экрана согласно ТЗ: `/shops/:shopId/growth/telegram`.
- **React Hook Form** — форма подключения интеграции (токен, `chat_id`, тумблер).
- **Axios** — HTTP-клиент; единый настроенный экземпляр (базовый URL, перехватчики ошибок, при необходимости — заголовки) и тонкая обёртка над ним, без прямого «размазывания» axios по компонентам.
- **Zod** — схемы валидации полей формы; согласование с React Hook Form (`@hookform/resolvers`).
- **Tailwind CSS** — стилизация; единые дизайн-токены (отступы, размеры контролов) через конфиг при необходимости.
- **TanStack Query (React Query)** — серверное состояние: загрузка статуса, инвалидация после сохранения, кэш и повторные запросы; мутация для отправки формы.
- **react-toastify** — уведомления об успехе и ошибках без привязки к разметке экрана.

---

## 2. Архитектура: Feature-Sliced Design (FSD)

Слои сверху вниз (зависимости только «вниз», не на соседние срезы внутри слоя без правил публичного API):

| Слой | Назначение в MVP |
|------|------------------|
| **`app`** | Провайдеры (QueryClient, Router, ToastContainer), глобальные стили Tailwind, точка входа. |
| **`pages`** | Страница экрана Telegram: сборка виджетов и секций, привязка к `shopId` из URL. |
| **`widgets`** | Крупные блоки экрана: «форма настроек», «блок статуса», при необходимости — композиция из features. |
| **`features`** | Прикладные действия: «сохранить настройки интеграции», при желании — отдельная feature для тестового заказа. |
| **`entities`** | Предметная модель UI: типы и маппинг данных статуса/ответа сохранения, при необходимости — маленькие презентационные кирпичи, не раздувая слой. |
| **`shared`** | Переиспользуемое: UI-kit (`Input`, `Button`), axios-api, хуки запросов, иконки, утилиты, конфиг API. |

Публичный вход среза — через `index.ts` (баррель), внутренности не импортировать извне напрямую.

---

## 3. Переиспользуемые компоненты (`shared/ui`)

- **`Input`** — обёртка над нативным полем ввода под Tailwind: label, ошибка, disabled, единый внешний вид; совместимость с `register` / `Controller` из React Hook Form.
- **`Button`** — варианты типа primary / secondary / danger (как минимум primary для «Сохранить»), состояния loading и disabled, тип `submit` для формы.

Оба компонента без бизнес-логики интеграции — только визуал и доступность (фокус, `aria-invalid` у полей с ошибкой).

---

## 4. Axios и кастомные хуки

- В **`shared/api`** разместить экземпляр axios и функции-«клиенты» по зонам ответственности (тонкие функции, возвращающие типизированные данные).
- Для **каждого обращения к API** завести **отдельный хук** в `shared/api` или в срезе `entities`/`features` (по правилу: если хук только про один сценарий страницы — ближе к feature; если общий — в `shared/api/hooks`). Имя хука отражает операцию (`useTelegramStatus`, `useTelegramConnectMutation` и т.д.).
- Внутри хуков использовать **React Query**: `useQuery` для чтения, `useMutation` для сохранения; axios вызывается только из этих хуков или из query-functions, не из компонентов формы напрямую.

Так унифицируются ошибки, отмена запросов и повторное использование.

---

## 5. Иконки

- Не вставлять SVG «сырым» JSX по месту использования.
- Вынести каждую иконку в отдельный компонент в **`shared/ui/icons`** (или `shared/icons`): единый контракт `className`, размер по умолчанию, `aria-hidden` где уместно.
- Импортировать на экране и в UI только готовые компоненты-иконки.

---

## 6. Форма и валидация

- Схемы Zod для полей формы (непустые строки для токена и `chat_id`, boolean для тумблера).
- Подключение через resolver к React Hook Form; сообщения ошибок пробрасывать в `Input`.
- После успешной мутации — инвалидация связанных query-ключей статуса и краткий toast.

---

## 7. Порядок внедрения (рекомендуемый)

1. Инициализировать проект (Vite + React + TS), подключить Tailwind, линтер TypeScript.
2. Развернуть каркас FSD: каталоги слоёв, `app` с провайдерами и роутом из ТЗ.
3. Реализовать `shared/api` (axios + типы) и хуки с React Query.
4. Добавить `Input`, `Button`, набор иконок в `shared/ui`.
5. Собрать `widgets` и `features`, затем страницу в `pages`.
6. Подключить react-toastify и сценарии ошибок (сеть, ответы с телом ошибки).
7. Задокументировать в корневом README команды установки и запуска фронтенда и способ указания базового URL API для окружений (см. корневой [README.md](../README.md), [doc/README.md](./README.md), [Что сделано — фаза 5](./Что%20сделано%20-%20фаза%205.md)).

## 8. Нефункциональные заметки

- Базовый URL API — через переменные окружения сборки (и при необходимости согласование с CORS на backend).
- Не дублировать в этом документе описание сущностей и HTTP-контрактов — они задаются ТЗ и реализацией API.
