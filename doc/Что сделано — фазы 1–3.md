# Что сделано: фазы 1–3

Документ для последующего анализа: фиксирует фактическое состояние репозитория после фаз **1** (модель и БД), **2** (клиент Telegram) и **3** (прикладные сервисы). Источник требований по смыслу — [ТЗ.md](./ТЗ.md), порядок работ — [План реализации MVP.md](./План%20реализации%20MVP.md).

---

## Фаза 1. Доменная модель и БД

- **Сущности Doctrine** в `src/Entity/`: `Shop`, `TelegramIntegration`, `ShopOrder` (таблица `orders`), `TelegramSendLog`; перечисление `TelegramSendStatus` (`SENT`, `FAILED`).
- **Связи и ограничения**: одна интеграция Telegram на магазин; журнал отправок с уникальной парой `(shop_id, order_id)`; каскады по ТЗ/миграции.
- **Миграция**: `migrations/Version20260404041733.php` — таблицы, внешние ключи, `CHECK` на допустимые значения `status` в `telegram_send_log`.
- **Репозитории**: `ShopRepository`, `ShopOrderRepository`, `TelegramIntegrationRepository`, `TelegramSendLogRepository` — в т.ч. методы для агрегатов экрана статуса (счётчики за период, последние `sentAt` по успеху и по любой попытке).
- **Сид**: консольная команда `app:seed-demo` (`src/Command/SeedDemoCommand.php`) — демо-магазин и набор тестовых заказов при пустой БД магазинов.
- **Окружение**: в корневом README и `config/services.yaml` зафиксирован параметр `telegram.use_real_api` / переменная `TELEGRAM_USE_REAL_API` (наследие фазы 0).

## Фаза 2. Интеграция Telegram (инфраструктура)

- **Контракт**: интерфейс `App\Telegram\TelegramClient` — метод отправки сообщения; ошибки через `TelegramSendException`.
- **Реализации**:
  - `HttpTelegramClient` — `POST` на Bot API `sendMessage` через Symfony `HttpClientInterface`, разбор `ok` / описание ошибки.
  - `MockTelegramClient` — без сети, фиксирует вызовы в памяти (`getRecordedCalls`, `resetRecordedCalls`) для разработки и тестов.
- **DI**: `TelegramClientFactory` + явная настройка в `config/services.yaml`: сервис `App\Telegram\TelegramClient` создаётся фабрикой по флагу `%telegram.use_real_api%`.
- **Слои**: прикладной код не зависит от HTTP-клиента напрямую — только от интерфейса `TelegramClient`.

## Фаза 3. Прикладные сервисы

Файлы: `src/Service/TelegramIntegrationUpsertService.php`, `src/Service/ShopOrderCreationService.php`, `src/Service/CreatedShopOrderResult.php`, `src/Enum/OrderNotificationDispatchStatus.php`, `src/Exception/*`, доработка `src/Repository/TelegramSendLogRepository.php`.

Цель фазы по плану: сценарии «настройка интеграции» и «создание заказа с уведомлением» без HTTP; идемпотентность по журналу; ошибка Telegram не отменяет уже сохранённый заказ. Слой контроллеров (фаза 4) будет вызывать эти сервисы и маппить исключения на 404/422.

### 1. `TelegramIntegrationUpsertService`

**Назначение:** одна операция на сценарий ТЗ «подключение / обновление Telegram для магазина» (upsert по `shopId`).

**Поведение:**

- Загружается магазин через `ShopRepository::find($shopId)`. Отсутствие записи → `ShopNotFoundException` с пояснением в сообщении (для последующего ответа HTTP 404).
- Строки `botToken` и `chatId` нормализуются через `trim`. Если после обрезки любая из них пустая → `InvalidTelegramIntegrationDataException` с текстом про обязательность полей (удобно сопоставить с HTTP 422 во фронте контроллера).
- Ищется существующая интеграция `TelegramIntegrationRepository::findOneByShopId`. Если нет — создаётся новая `TelegramIntegration`, выставляется связь `setShop($shop)` и `persist`. Если есть — обновляются поля на управляемой сущности.
- В любом случае выставляются `botToken`, `chatId`, `enabled` (тумблер из payload будущего API).
- Один вызов `flush()` в конце: Doctrine обновляет временные метки через колбэки сущности (`PrePersist` при создании, `PreUpdate` при изменении — см. `TelegramIntegration`).

**Результат:** возвращается актуальная сущность `TelegramIntegration` (для сериализации в ответе `connect` на фазе 4).

**Замечание:** сервис сам не ходит в Telegram и не валидирует токен у Bot API — только непустые строки и существование магазина.

---

### 2. `ShopOrderCreationService::createAndNotify`

**Назначение:** сценарий ТЗ «создание заказа (эмуляция)» + уведомление в Telegram при включённой интеграции.

**Вход:** `shopId`, `number`, `total` (строка, дальше нормализуется), опционально `customerName`.

**Шаг 1 — заказ в БД до внешнего вызова**

- Проверка магазина: нет → `ShopNotFoundException`.
- Создаётся `ShopOrder`: привязка к `Shop`, номер, сумма через `normalizeTotal` (приведение к строке с двумя знаками после запятой для `NUMERIC` в БД), имя клиента через `normalizeCustomerName` (`trim`, пустая строка → `null`).
- `persist` + `flush`: заказ получает `id`. Это сознательный порядок: к моменту вызова Telegram заказ уже записан; сбой доставки не откатывает транзакцию заказа вместе с «ещё не отправленным»(notification) — заказ остаётся в таблице `orders`.

**Шаг 2 — `dispatchTelegramIfNeeded` (ветвление и идемпотентность)**

- Нет `orderId` после flush (защитный код) → статус **`skipped`**.
- Нет строки интеграции для магазина **или** `enabled === false` → **`skipped`**, **без** вызова `TelegramClient`, **без** строки в `telegram_send_log` (уведомление не предполагалось).
- Есть запись в журнале для пары `(shopId, orderId)` (`findOneByShopIdAndOrderId`) → **`skipped`**, клиент не вызывается (повторная доставка для того же заказа не выполняется).

**Шаг 3 — текст сообщения**

- Формат строго по ТЗ:  
  `Новый заказ {number} на сумму {total} ₽, клиент {customerName}`  
- Сумма из поля заказа: отображение без лишних нулей после запятой (`rtrim` нулей и точки).  
- Если имя клиента `null` или пустая строка — подставляется **`не указан`**.

**Шаг 4 — вызов клиента и журнал**

- Успешный `TelegramClient::sendMessage(botToken, chatId, text)` → запись в журнал со статусом **`SENT`** (см. ниже). Публичный статус сценария → **`sent`** (`OrderNotificationDispatchStatus::SENT`).
- Перехватывается только `TelegramSendException` (контракт фазы 2). Сообщение об ошибке сохраняется в колонку `error` журнала, статус строки **`FAILED`**, публичный результат → **`failed`**.
- Любая другая непойманная ошибка в этом методе пробросится выше (на фазе 4 решится политикой контроллера / логированием).

**Запись журнала (`tryPersistSendLog`) — важно для анализа**

- Не используется `flush()` новой сущности `TelegramSendLog` через ORM для этой вставки, чтобы избежать сценария «конфликт уникального ключа → исключение при commit → закрытие EntityManager» в Doctrine при гонках.
- Выполняется прямой SQL:  
  `INSERT INTO telegram_send_log (...) VALUES (...) ON CONFLICT (shop_id, order_id) DO NOTHING`  
  Параметры: текст сообщения, `status` (`SENT`/`FAILED` как строка enum сущности), `error` (или SQL `NULL`), `sent_at` в ISO 8601 (`format('c')`), внешние ключи `shop_id` / `order_id`.
- Если вставлена **ровно одна** строка (`executeStatement` вернул число строк > 0) — метод возвращает `true`, и наружу идёт `sent` или `failed` в зависимости от ветки `try/catch`.
- Если конфликт по `(shop_id, order_id)` и строка **не** вставлена — возвращается `false`, наружу отдаётся **`skipped`**: трактовка «запись журнала уже занята другим потоком или предыдущей логикой», повторный ответ не заявляет успех отправки.

**Итоговый объект:** `CreatedShopOrderResult` — управляемый (или после дальнейших запросов — отсоединённый) `ShopOrder` и значение `OrderNotificationDispatchStatus` для маппинга в JSON поля вроде `notificationStatus` / `telegramStatus` на фазе 4.

### 3. Вспомогательные типы и репозиторий

- **`OrderNotificationDispatchStatus`** (`src/Enum/OrderNotificationDispatchStatus.php`): строковые значения для API — `sent`, `failed`, `skipped` (в ТЗ для ответа создания заказа).
- **`CreatedShopOrderResult`**: пара «заказ + статус уведомления»; контроллер не обязан знать внутренности `dispatchTelegramIfNeeded`.
- **`ShopNotFoundException`**, **`InvalidTelegramIntegrationDataException`**: границы доменных ошибок для маппинга на HTTP (ожидаемо 404 и 422).
- **`TelegramSendLogRepository::findOneByShopIdAndOrderId`**: выборка сущности журнала по магазину и заказу; используется для раннего выхода без вызова Telegram (идемпотентность в типичном случае без гонки).
