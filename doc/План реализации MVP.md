# План реализации MVP (интеграция Telegram для Posiflora)

Источник требований: [О проекте.md](./ТЗ.md). Контракты HTTP, поля JSON и DDL — только там; ниже пошаговый порядок работ и архитектурные решения под текущий стек (Symfony 8, Doctrine, PostgreSQL, Docker Compose).

## Цель MVP

Магазин подключает Telegram (токен и chat_id), при новом заказе уходит уведомление, есть экран настроек и статуса, отправка идемпотентна и не отменяет создание заказа при сбое Telegram.

---

## Фаза 0. Подготовка репозитория и окружения

1. Убедиться, что `docker compose up -d` поднимает PostgreSQL и `DATABASE_URL` в `.env` / `.env.local` совпадает с контейнером (см. [README.md](./README.md)).
2. Ввести переменную окружения для переключения реальной отправки и мока (например флаг в `.env`), задокументировать в корневом README.
3. При необходимости зафиксировать локальные проверки качества (PHPStan, CS Fixer и т.д.) в том виде, как принято в репозитории.

---

## Фаза 1. Доменная модель и БД

1. Сущности Doctrine в `src/Entity/` по схеме из ТЗ: магазины, интеграция Telegram (одна на магазин), заказы, журнал отправок с уникальностью пары (магазин, заказ).
2. Миграции Doctrine: таблицы, внешние ключи, ограничения (в т.ч. допустимые значения статуса отправки, если СУБД поддерживает).
3. Репозитории: запросы для экрана статуса — последняя успешная/любая отправка по времени, счётчики успехов и ошибок за скользящие 7 дней.
4. Сид: минимум один магазин и несколько тестовых заказов (DataFixtures или консольная команда). Секреты реального бота в репозиторий не коммитить.

---

## Фаза 2. Интеграция Telegram

1. Интерфейс `TelegramClient` в кодовой базе: абстракция надметодом отправки сообщения; контроллеры и репозитории о нём не знают.
2. Реализация через Symfony HttpClient на официальный Bot API (`sendMessage`).
3. Мок-реализация без сети для тестов и для локального режима; выбор реализации через DI (`services.yaml`, условия окружения).
4. Зависимости прикладного слоя только от интерфейса, не от HTTP-клиента напрямую.

---

## Фаза 3. Прикладные сервисы

1. Сервис настройки интеграции: upsert по магазину, валидация непустых токена и chat_id, обновление флага активности и меток времени.
2. Сценарий создания заказа: сохранить заказ; если интеграция выключена или отсутствует — не вызывать Telegram, вернуть согласованный статус в ответе (в ТЗ — `skipped` среди вариантов).
3. Перед отправкой проверить журнал: если для пары (магазин, заказ) запись уже есть — не слать повторно и не плодить строки.
4. Вызвать `TelegramClient`; при успехе и при ошибке записать строку в журнал с соответствующим статусом; **ошибка Telegram не откатывает транзакцию заказа** (заказ остаётся сохранённым).

Слои: обработчик HTTP → сервис → репозиторий + `TelegramClient`.

## Фаза 4. Backend HTTP

1. Три маршрута из ТЗ: подключение интеграции, создание заказа (эмуляция), статус интеграции — с префиксом `shops/{shopId}` как в документе.
2. DTO и Symfony Validator для тел запросов; JSON-ответы; для статуса — маскирование chat_id при отдаче наружу.
3. Коды ответов: не найден магазин — 404; невалидное тело — 422.
4. При отдельном origin у фронта — минимальная настройка CORS для разработки.

## Фаза 7. Тесты

Минимум три сценария из ТЗ:

1. Включённая интеграция, мок клиента фиксирует вызов; после создания заказа в журнале статус успешной отправки.
2. Повтор не приводит к второй записи в журнале и к повторному вызову клиента (идемпотентность).
3. Клиент имитирует ошибку; в журнале неуспех, запись заказа в БД присутствует.

При желании добавить тесты upsert настроек и агрегатов статуса за 7 дней. Использовать `WebTestCase` и/или `KernelTestCase` в зависимости от того, что тестируется — HTTP или только сервисы.

## Риски и допущения MVP

- **Безопасность**: токен в БД в открытом виде для MVP — отразить в README; показывать пользователю маскированный chat_id/токен где уместно.
- **Гонки**: уникальный ключ в журнале предотвращает дубли; при конфликте вставки обработать как «уже отправлено».
- **Асинхронность**: в MVP отправка в том же запросе; дальнейшее развитие — Messenger, ретраи, мониторинг — тема для обсуждения после MVP.
