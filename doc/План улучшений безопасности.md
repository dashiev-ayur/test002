# План пошаговых улучшений безопасности

Документ фиксирует последовательность работ по усилению защиты backend. Порядок отражает приоритет: сначала закрывают критические риски, затем инфраструктура и hardening.

---

## Этап 1. Аутентификация и границы доступа

1. Определить модель доверия: кто может вызывать API (только бэкенд CRM, отдельный сервис, публичный ключ магазина и т.д.).
2. Ввести механизм проверки запросов на защищённых маршрутах (например, статический API‑ключ в заголовке, HMAC подпись тела запроса, mutual TLS — в зависимости от модели из п.1).
3. Настроить Symfony Security: отдельный firewall для API, кастомный authenticator или entry point под выбранную схему.
4. Закрыть изменяющие операции (подключение интеграций, создание заказов) до успешной аутентификации; чтение статуса — по тем же правилам или более строгим, если данные чувствительные.
5. Убедиться, что секреты для проверки подлинности хранятся только в переменных окружения / секрет‑хранилище, не в репозитории.

---

## Этап 2. Авторизация (привязка к магазину)

1. После идентификации клиента проверять, что операция разрешена **именно для указанного магазина** (ключ или токен не даёт доступ к чужим идентификаторам).
2. Исключить сценарий «один глобальный ключ на всё» без дополнительных проверок, если в системе несколько независимых магазинов.

---

## Этап 3. Секреты и конфигурация окружения

1. Убрать из продакшен‑деплоя значения по умолчанию для секретов приложения и БД; задавать их явно при выкатке.
2. Проверить, что в репозиторий не попадают реальные `.env` с продакшен‑значениями; использовать `.env.local` / CI secrets.
3. Регулярно ротировать `APP_SECRET` и ключи API при компрометации или по регламенту.

---

## Этап 4. CORS

1. В продакшене зафиксировать список доверенных origin; не использовать `*` и не отражать произвольный `Origin` без жёсткого whitelist.
2. Если позже понадобятся cookie или чувствительные заголовки — явно настроить `Access-Control-Allow-Credentials` и согласовать это со списком origin.

---

## Этап 5. Ограничение нагрузки и злоупотреблений

1. Ввести rate limiting на создание заказов и на операции, трогающие внешние API (уведомления), по IP и/или по идентификатору клиента после внедрения auth.
2. При необходимости добавить очередь или backoff при ошибках внешнего провайдера, чтобы не зацикливать запросы.

---

## Этап 6. Валидация входных данных

1. Ужесточить правила для токена бота и идентификатора чата под допустимые форматы провайдера (длина, символы), чтобы отсекать очевидный мусор до записи в БД.
2. Для денежных и строковых полей продумать верхние границы длины и допустимые символы, согласованные с бизнес‑логикой.

---

## Этап 7. Логи и наблюдаемость

1. В продакшене снизить детализацию логов по умолчанию для успешного трафика; подробности — только на уровне ошибок и без утечки секретов.
2. Не логировать полные токены, пароли, полные тела ответов провайдеров с секретами; при отладке — маскирование как в уже используемой утилите маскирования.
3. Привести ответы об «объекте не найден» к единообразию, если нужно усложнить перечисление идентификаторов (детали — только в серверных логах).

---

## Этап 8. Транспорт и инфраструктура

1. Терминировать TLS на границе (балансировщик, reverse proxy) в реальной среде; не опираться только на HTTP внутри незадокументированных доверенных сетей.
2. Добавить в конфигурацию веб‑сервера базовые заголовки безопасности (например, ограничение framing), где это уместно для вашего типа ответов.
3. Ограничить доступ к административным портам БД и к internal‑сервисам файрволом / сетевыми политиками.

---

## Этап 9. Процесс и зависимости

1. Подключить проверку уязвимостей зависимостей (`composer audit` или аналог в CI).
2. Обновлять PHP и пакеты по расписанию; фиксировать минимальные версии в политике релизов.

---

## Этап 10. Проверка после внедрения

1. Прогнать интеграционные тесты с отказом в доступе без учётных данных и с неверным магазином.
2. Ручная или автоматическая проверка: недоступность защищённых маршрутов без auth, корректные CORS в прод‑конфиге, отсутствие секретов в образах и логах.

---

*Документ можно дополнять по мере появления новых интеграций и требований комплаенса.*
